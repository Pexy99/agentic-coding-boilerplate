# Project Workflow

## 1. 개요
이 프로젝트는 작은 팀과 AI 에이전트가 같은 문맥으로 일하기 위한 문서 기반 워크플로우를 사용합니다. 문서는 작업을 돕기 위한 것이며, 작은 변경까지 불필요하게 무겁게 만들지 않습니다.

## 2. 핵심 원칙 (Core Principles)
1. **Docs as Shared Context**: 제품 맥락, 기술 선택, 작업 규칙은 `docs/`에 둡니다.
2. **Plan as Task Status**: 트랙 작업의 진행 상태는 `docs/tracks/<id>/plan.md`에 기록합니다.
3. **Branch Before Editing**: 문서나 코드를 수정하기 전에 `main`이 아닌 작업 브랜치로 분리합니다.
4. **Sync Before Work**: 주요 작업 시작 전과 PR 생성 전 `git fetch origin main`으로 원격 상태를 확인합니다.
5. **Keep It Small**: `spec.md`와 `plan.md`는 짧고 실행 가능한 수준으로 유지합니다.
6. **Prefer Automated Verification**: 코드 변경은 자동화 테스트를 우선하고, 어렵다면 이유와 수동 검증 결과를 남깁니다.
7. **Declare Before Acting**: 에이전트는 파일 수정 전 사용할 흐름, 브랜치 상태, 원격 동기화, 검증 방법을 먼저 알립니다.
8. **Constrain Structure**: 프로젝트 구조는 자유롭게 만들지 않고, `docs/tech-stack.md`와 팀 규칙에 명시된 구조를 따릅니다.

## 3. 작업 크기 기준

### Level 1: 작은 수정
오타, 문서 정리, 간단한 스타일 수정처럼 영향 범위가 작은 작업입니다.
- 트랙 없이 바로 수정할 수 있습니다.
- 커밋 또는 PR 설명에 변경 이유를 남깁니다.

### Level 2: 일반 작업
기능 추가, 버그 수정, API 변경, 화면 변경처럼 팀원이 맥락을 알아야 하는 작업입니다.
- `start-track` 스킬로 트랙을 만듭니다.
- 스킬이 `spec.md`와 `plan.md`를 짧게 작성하거나, 정보가 부족하면 먼저 질문합니다.
- `docs/tracks.md`에 활성 트랙으로 등록합니다.
- 트랙 ID는 `<type>_<short_name>_<YYYYMMDD>` 형식을 사용합니다. 예: `feat_login_20260419`

### Level 3: 큰 결정
DB, 인증, 배포, 프레임워크, 큰 구조 변경처럼 되돌리기 어려운 작업입니다.
- Level 2 절차를 따릅니다.
- 필요하면 `docs/adr/`에 결정 기록을 남깁니다.
- 기술 선택이 바뀌면 `docs/tech-stack.md`를 업데이트합니다.

## 4. 트랙 작업 흐름

1. `docs/tracks.md`에서 활성 트랙을 확인합니다.
2. 해당 트랙의 `spec.md`와 `plan.md`를 읽습니다.
3. 시작한 항목은 `plan.md`에서 `[~]`로 표시합니다.
4. 구현하고 검증합니다.
5. 완료한 항목은 `plan.md`에서 `[x]`로 표시합니다.
6. 충족된 완료 기준은 `spec.md`에서 `[x]`로 표시합니다.
7. 가능하면 완료 항목 옆에 커밋 해시를 남깁니다.

## 5. 검증 기준

작업 완료 전 다음을 자가 점검합니다:
- [ ] 관련 자동화 테스트 통과 여부
- [ ] 테스트를 추가/수정하지 않았다면 이유 기록
- [ ] 자동 검증이 어렵다면 수동 확인 방법과 결과 기록
- [ ] UI E2E를 남발하지 않고, 필요한 경우 핵심 happy path에만 제한했는지 확인
- [ ] UI 변경이면 사람이 따라할 수 있는 검증 체크리스트를 남겼는지 확인
- [ ] 관련 코드 스타일 가이드 준수 (`docs/code_styleguides/` 참조)
- [ ] 필요한 문서 업데이트 여부
- [ ] `plan.md`와 `spec.md` 상태 동기화 여부

## 6. 구조 규칙

- 루트에는 설정 파일, 문서, 빌드 진입점처럼 전역 파일만 둡니다.
- 앱 코드는 가능하면 `src/` 아래에 둡니다.
- 새 프레임워크 프로젝트는 가능하면 공식 CLI/스캐폴더로 시작합니다.
- 구조 규칙이 바뀌면 `docs/tech-stack.md`와 관련 스타일가이드를 함께 업데이트합니다.

## 7. 테스트 철학

- 결정론적인 계층은 자동화를 우선합니다.
  - 예: 단위 테스트, 순수 함수 테스트, API 입력/출력 테스트, 데이터 파이프라인 검증
- UI E2E는 최소화합니다.
  - 핵심 happy path만 자동화하고, 나머지는 수동 검증 체크리스트를 남깁니다.
- 에이전트는 UI 전체를 무겁게 E2E로 덮기보다 사람이 따라할 검증 절차를 제안하는 쪽을 우선합니다.
- 최종 UI 확인과 병합 책임은 사람이 집니다.

## 8. 선택 기록

- `git notes`는 선택 사항입니다. PR 설명과 `plan.md`만으로 충분하면 사용하지 않아도 됩니다.
- ADR은 큰 기술 결정에만 작성합니다.
- 체크포인트 커밋은 긴 작업에서 필요할 때만 만듭니다.

## 9. 이름 규칙

트랙 ID, 브랜치, 커밋 메시지는 같은 type을 사용합니다.

- 사용 가능한 type: `feat`, `fix`, `docs`, `refactor`, `test`, `chore`
- 트랙 ID: `feat_login_20260419`
- 브랜치: `feat/login_20260419`
- 커밋: `feat: add login form`
